🧪 الاختبارات الآلية (Testing): كيف نضمن أن كودنا يعمل للأبد؟

"المبرمج المحترف لا يختبر الكود يدوياً في كل مرة؛ بل يكتب كوداً يختبر كوده!"

🔍 ما هي الاختبارات (Tests)؟

الاختبارات هي عبارة عن ملفات برمجية نكتبها بلغة PHP (في حالة Laravel) تقوم بمحاكاة سلوك المستخدم الحقيقي. تفتح صفحات، تضغط على أزرار، ترسل بيانات، وتتأكد أن النتيجة صحيحة. كل هذا يحدث بشكل آلي خلال ثوانٍ.

تخيل أنك مطور في فريق FlowPilot. طلب منك مديرك إضافة ميزة "تصفية المشاريع حسب التاريخ". بعد كتابة الكود، كيف تتأكد أن الميزة الجديدة لم تفسد شيئاً قديماً؟ الإجابة: تشغل الاختبارات.

⚡ مصطلح مهم: Test Suite

هو مجموعة كل ملفات الاختبار في مشروعك. كلما زادت، زادت ثقتك بجودة الكود.

📚 لماذا نكتب الاختبارات؟ (القيمة الحقيقية)

🛡️

1. منع التراجع (Regression Protection)

تضمن أن أي تعديل جديد لا يكسر شيئاً كان يعمل سابقاً. مثلاً: إضافة حقل "تاريخ الاستحقاق" للمهام لا يجب أن يعطل عرض قائمة المهام.

✅

2. الثقة عند النشر (Deployment Confidence)

قبل رفع الكود إلى سيرفر الإنتاج (Production Server)، تشغل الاختبارات. إذا كانت كلها خضراء ✅، يمكنك النوم مرتاحاً.

📖

3. توثيق حي (Living Documentation)

ملف الاختبار يشرح بالضبط كيف يفترض أن تعمل كل ميزة. أي مبرمج جديد يقرأ الاختبارات سيفهم النظام بسرعة.

⏱️

4. توفير الوقت على المدى البعيد

كتابة اختبار تأخذ 10 دقائق الآن، لكنها قد توفر لك ساعات من التصحيح اليدوي بعد شهر.

🚗 تشبيه واقعي: فحص السيارة في المصنع

تخيل أنك تملك مصنع سيارات. كل يوم تنتج 100 سيارة.

👨‍🔧 الطريقة اليدوية

موظف يركب كل سيارة، يدور بالمفتاح، يضغط الفرامل، يتأكد من الأنوار. لكل سيارة 20 دقيقة. هذا بطيء وممل وعرضة للخطأ.

🤖 الطريقة الآلية (Testing)

روبوت مبرمج يختبر 100 سيارة في 5 دقائق. يفحص المحرك، الفرامل، الأنوار، المكيف. يعطي تقريراً: 100 سيارة سليمة ✅ أو السيارة رقم 7 فيها عطل ❌.

الخلاصة: Testing هو "الروبوت" الذي يفحص تطبيقك نيابة عنك.

🔗 كيف نستخدم Testing في FlowPilot؟

في مشروع FlowPilot، سنكتب اختبارات تغطي السيناريوهات التالية:

  • زيارة صفحة المشاريع: نتأكد أن المستخدم المسجل يستطيع رؤية قائمة مشاريعه.
  • إنشاء مشروع جديد: نتأكد أن إرسال النموذج ينشئ مشروعاً في قاعدة البيانات.
  • التحكم بالصلاحيات (Policy): نتأكد أن المستخدم "أ" لا يستطيع تعديل مشروع المستخدم "ب".
  • تسجيل الخروج: نتأكد أن زر تسجيل الخروج يعيد التوجيه إلى صفحة تسجيل الدخول.

🛠️ التطبيق العملي خطوة بخطوة

الخطوة 1: إنشاء ملف الاختبار

نستخدم أمر Artisan لإنشاء ملف اختبار Feature مخصص للمشاريع:

php artisan make:test ProjectTest

💡 شرح: هذا الأمر يُنشئ ملفاً جديداً في المسار tests/Feature/ProjectTest.php. اسم الملف يتبع PascalCase (كل كلمة تبدأ بحرف كبير).

الخطوة 2: كتابة اختبار فتح صفحة المشاريع

نفتح الملف الذي تم إنشاؤه ونكتب أول دالة اختبار:

public function test_projects_page_is_accessible(): void

{

    $response = $this->get('/projects');

    $response->assertStatus(200);

}

🔍 شرح الكود سطراً سطراً:

public function test_projects_page_is_accessible(): void — تعريف دالة عامة. كل دالة تبدأ بكلمة test_ يعتبرها Laravel اختباراً.

$response = $this->get('/projects'); — نستخدم $this->get() لإرسال طلب HTTP من نوع GET إلى الرابط /projects. النتيجة تحفظ في $response.

$response->assertStatus(200); — نتأكد أن الاستجابة تحمل رمز الحالة 200، أي أن الصفحة فتحت بنجاح.

الخطوة 3: اختبار إنشاء مشروع (مع مصادقة)

لنكتب اختباراً يتأكد من أن المستخدم المسجل يمكنه إنشاء مشروع:

public function test_authenticated_user_can_create_project(): void

{

    $user = User::factory()->create();

    $this->actingAs($user);

    $response = $this->post('/projects', [

        'name' => 'مشروع تجريبي',

        'description' => 'وصف المشروع',

    ]);

    $response->assertStatus(302);

    $this->assertDatabaseHas('projects', [

        'name' => 'مشروع تجريبي',

    ]);

}

🔍 التحليل:

User::factory()->create() — ينشئ مستخدم وهمي باستخدام Factory (مصنع بيانات).

$this->actingAs($user) — يسجل دخول المستخدم الوهمي للنظام.

$this->post('/projects', [...]) — يرسل طلب POST (مثل إرسال نموذج).

assertStatus(302) — نتأكد من التوجيه (Redirect) بعد الإنشاء الناجح.

assertDatabaseHas('projects', [...]) — نتأكد أن البيانات حُفظت فعلاً في قاعدة البيانات.

الخطوة 4: اختبار صلاحيات الوصول (Policy)

نتأكد أن المستخدم لا يستطيع تعديل مشروع ليس ملكه:

public function test_user_cannot_update_others_project(): void

{

    $owner = User::factory()->create();

    $otherUser = User::factory()->create();

    $project = Project::factory()->create([

        'user_id' => $owner->id

    ]);

    $this->actingAs($otherUser);

    $response = $this->put("/projects/{$project->id}", [

        'name' => 'اسم معدل'

    ]);

    $response->assertStatus(403);

}

🔍 التحليل:

assertStatus(403) — 403 يعني "ممنوع" (Forbidden). نتأكد أن النظام منع المستخدم.

لاحظ: هذا الاختبار يعتمد على وجود Policy في Laravel تتحكم بصلاحيات التعديل.

الخطوة 5: تشغيل جميع الاختبارات

نشغل الأمر التالي في التيرمنال:

php artisan test

ستظهر شاشة ملونة. إذا كانت كل الاختبارات خضراء ✅، فتهانينا! كودك آمن.

الخطوة 6: تشغيل اختبار واحد فقط

لتوفير الوقت، يمكنك تشغيل اختبار معين باسم الملف:

php artisan test --filter ProjectTest

الـ --filter يسمح بتصفية الاختبارات التي تحتوي على الكلمة المحددة.

🎯 النتيجة المتوقعة

بعد تنفيذ الخطوات السابقة، عند تشغيل php artisan test يجب أن ترى:

PASS Tests\Feature\ProjectTest

✓ projects page is accessible

✓ authenticated user can create project

✓ user cannot update others project

Tests: 3 passed

كل كلمة "PASS" أو "✓" تعني أن الاختبار نجح. إذا ظهر "FAIL" أو "✗" فهناك خطأ في الكود.

⚠️ الأخطاء الشائعة التي قد تواجهك

🚫 الخطأ 1: لا يوجد جدول في قاعدة البيانات

إذا ظهر خطأ Base table or view not found، فهذا يعني أن جدول projects غير موجود.

الحل: شغل php artisan migrate أولاً لإنشاء الجداول.

🚫 الخطأ 2: مصادقة مطلوبة (Unauthenticated)

إذا ظهر خطأ Expected response status code 200 but received 302، يعني أن الصفحة تعيد التوجيه إلى تسجيل الدخول.

الحل: استخدم $this->actingAs($user) قبل محاولة زيارة الصفحة المحمية.

🚫 الخطأ 3: إملاء خاطئ لأسماء الحقول

إذا ظهر SQLSTATE[HY000]: General error، قد يكون اسم حقل في قاعدة البيانات خطأ.

الحل: تأكد أن أسماء الحقول تطابق تماماً ما في ملف migration.

🚫 الخطأ 4: اختبار وحيد يتسبب بفشل باقي الاختبارات

أحياناً اختبار يعدل قاعدة البيانات ويؤثر على الاختبار التالي.

الحل: استخدم use RefreshDatabase; في بداية كلاس الاختبار لإعادة تعيين قاعدة البيانات بعد كل اختبار.

📋 الملخص النهائي

📌 ما تعلمناه:

  • Testing = كود يختبر كود آخر آلياً
  • نوعان: Feature tests (للميزات) و Unit tests (للأجزاء الصغيرة)
  • أداة Pest هي الافتراضية في Laravel 12
  • الأمر php artisan make:test ينشئ ملف اختبار
  • الدوال تبدأ بـ test_ ليتعرف عليها Laravel
  • assertStatus() و assertDatabaseHas() دوال تحقق
  • الـ Policy تتحكم بالصلاحيات ونختبرها بسيناريوهات

💡 تذكر دائماً:

  • الاختبارات توفر الوقت على المدى البعيد
  • الاختبارات توثق الكود لمن يأتي بعدك
  • الاختبارات تمنحك الثقة للنشر في الإنتاج
  • ابدأ باختبارات بسيطة ثم زد التعقيد

"الكود بدون اختبارات هو كود ميت لم يولد بعد."
