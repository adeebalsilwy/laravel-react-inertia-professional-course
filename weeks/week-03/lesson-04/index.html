قراءة البيانات بكفاءة وتحسين الاستعلامات

"التطبيق البطيء يخسر المستخدمين. تعلم كيف تجعل استعلاماتك أسرع بثلاث طرق."

لماذا أداء قاعدة البيانات مهم جداً؟

تخيل تطبيق FlowPilot مع 1000 مستخدم، كل مستخدم لديه 10 مشاريع، وكل مشروع لديه 20 مهمة. الآن تخيل أن كل مرة يفتح فيها المستخدم صفحة المشاريع، يتم تنفيذ 1000 استعلام MySQL. النتيجة: صفحة تفتح بعد 10 ثوانٍ!

المستخدم العادي يتوقع أن تفتح الصفحة في أقل من 2-3 ثوانٍ. إذا كان تطبيقك بطيئاً، سيتجه المستخدمون إلى تطبيق آخر. لذا تحسين الاستعلامات ليس "تطرياً" - إنه ضرورة.

في هذا الدرس، سنتعلم 6 تقنيات قوية لتحسين أداء استعلامات Eloquent. يمكنك تطبيقها فوراً في FlowPilot.

100 استعلام لصفحة واحدة
زمن التحميل: 8 ثوانٍ
تجربة مستخدم سيئة

👍 بعد التحسين

3 استعلامات لصفحة واحدة
زمن التحميل: 0.5 ثانية

Eager Loading: جلب العلاقات مقدماً بـ ::with()

كما تعلمنا في الدرس السابق عن N+1، استخدام ::with() يقلل عدد الاستعلامات بشكل كبير. لكنه ليس مجرد حل لـ N+1. يمكن لـ ::with() القيام بأمور أكثر تقدماً.

جلب علاقات متداخلة (Nested):

// جلب المشاريع + المستخدم + المهام + المسؤول عن كل مهمة $projects = Project::with([ 'user', // صاحب المشروع 'tasks', // مهام المشروع 'tasks.assignee', // المسؤول عن كل مهمة (علاقة متداخلة) 'members', // أعضاء الفريق ])->get(); // استعلام واحد فقط + 4 استعلامات JOIN = 5 استعلامات بدلاً من N+1

constrain مع with (شروط على العلاقات):

// جلب المشاريع والمهام النشطة فقط $projects = Project::with([ 'tasks' => function ($query) { $query->where('status', 'in_progress') // فقط المهام النشطة ->orderBy('priority', 'desc'); // الأعلى أولوية أولاً }, ])->get();

withCount (جلب عدد العناصر في العلاقة):

$projects = Project::withCount([ 'tasks', // $project->tasks_count 'tasks as completed_tasks_count' => function ($query) { $query->where('status', 'completed'); // $project->completed_tasks_count }, 'members', // $project->members_count ])->get();

هذا يضيف أعمدة إضافية للنتيجة مثل tasks_count و members_count - بدون استعلامات إضافية!

Lazy Loading vs Eager Loading: متى تستخدم كل منهما؟

🐌 Lazy Loading (افتراضي)

يجلب العلاقة فقط عند استخدامها. إذا لم تستخدم العلاقة، لا استعلام إضافي.

مفيد عندما: لا تعرف إذا كنت ستحتاج العلاقة أم لا.

خطير عندما: تستخدم العلاقة في حلقة (يسبب N+1).

$p = Project::find(1); // استعلام واحد
echo $p->user->name; // استعلام إضافي هنا

⚡ Eager Loading (مع ::with)

يجلب العلاقة مسبقاً مع الاستعلام الأصلي باستخدام JOIN.

مفيد عندما: تعلم أنك ستحتاج العلاقة (مثلاً في صفحة listing).

أفضل أداء عندما: تستخدم العلاقة عدة مرات.

$ps = Project::with('user')->get(); // 2 استعلامات
foreach($ps as $p) { echo $p->user->name; } // لا استعلامات إضافية

💡 قاعدة بسيطة:

إذا كنت تعلم أنك ستستخدم العلاقة، استخدم ::with(). إذا كنت غير متأكد، استخدم Lazy Loading لكن ليس في حلقة.

Query Scopes: دوال استعلام قابلة لإعادة الاستخدام

Query Scope هي دالة في الـ Model تسمح لك بإضافة شروط استعلام بشكل متكرر دون تكرار الكود. هناك نوعان:

1. Local Scope (نطاق محلي) - يبدأ بـ scope

// في Project Model public function scopeActive($query) { return $query->where('status', 'active'); } public function scopeWithUserAndTasks($query) { return $query->with(['user', 'tasks']); }

استخدام الـ Scope:

$activeProjects = Project::active() // استدعاء الـ scopeActive ->withUserAndTasks() // استدعاء الـ scopeWithUserAndTasks ->latest() // إضافات أخرى ->get();

ملاحظة: عند الاستدعاء، نحذف scope من اسم الدالة. scopeActive يصبح active().

Scope مع معاملات (Parameters):

// تعريف public function scopeStatus($query, $status) { return $query->where('status', $status); } // استخدام $pendingProjects = Project::status('pending')->get();

2. Global Scope (تطبق على كل استعلام)

تطبق تلقائياً على جميع استعلامات الـ Model. مثلاً، إخفاء المشاريع المحذوفة (هذا ما يفعله SoftDeletes تلقائياً).

// تسجيل Global Scope في Boot Method protected static function booted(): void { static::addGlobalScope('active', function ($builder) { $builder->where('status', 'active'); }); }

⚠️ تحذير: Global Scopes خطيرة لأنها تطبق على كل استعلام. قد تسبب نتائج غير متوقعة. استخدمها بحذر.

::when(): الاستعلام الشرطي الذكي

تخيل أن لديك صفحة تبحث عن المشاريع. المستخدم قد يختار فلترة حسب الحالة أو لا. كيف تكتب استعلاماً يتعامل مع الحالتين؟

بدون ::when()، كنت ستكتب إذا...آخر... (if/else). مع ::when()، نكتب شرطاً واحداً أنيقاً:

$status = request('status'); // قد يكون null أو 'active' أو 'completed' $search = request('search'); // قد يكون null أو نص بحث $projects = Project::with('user') ->when($status, function ($query) use ($status) { // إذا كان $status موجوداً (غير فارغ)، أضف شرط where return $query->where('status', $status); }) ->when($search, function ($query) use ($search) { // إذا كان $search موجوداً، أضف شرط LIKE return $query->where('name', 'like', "%{$search}%"); }) ->latest() ->paginate(10); // تقسيم النتائج إلى صفحات

💡 شرح ::when():

الـ معامل الأول هو الشرط. إذا كان true (غير null، غير false، غير '')، يتم تنفيذ الـ Closure في المعامل الثاني. الفكرة: نبني الاستعلام ديناميكياً حسب مدخلات المستخدم.

Chunking: تقسيم النتائج الكبيرة إلى أجزاء

ماذا لو كان لديك 100,000 مشروع واستخدمت ::all()؟ سيحاول PHP تحميل جميع المشاريع في الذاكرة دفعة واحدة. هذا سيستهلك ذاكرة هائلة وقد يوقف التطبيق!

Chunking يقسم النتائج إلى أجزاء صغيرة (مثلاً 100 سجل في كل مرة) ويعالجها واحدة تلو الأخرى. هذا يقلل استهلاك الذاكرة بشكل كبير.

استخدام chunk():

// معالجة 100 مشروع في كل دورة Project::chunk(100, function ($projects) { // $projects تحتوي على 100 مشروع فقط foreach ($projects as $project) { // افعل شيئاً مع كل مشروع Log::info("معالجة المشروع: {$project->name}"); } });

هذا يعالج 100,000 مشروع في 1,000 دورة، كل دورة تحمل 100 مشروع فقط. استخدام الذاكرة ثابت ومنخفض.

chunkById للعمليات الكبيرة:

// مثالي لتحديث كميات كبيرة من البيانات Project::chunkById(100, function ($projects) { foreach ($projects as $project) { $project->update(['status' => 'archived']); } });

Caching: حفظ الاستعلامات في الذاكرة المؤقتة

الـ Cache هو مكان سريع التخزين (مثل الملفات أو Redis). الفكرة: إذا كان هناك استعلام ثقيل يتكرر كثيراً (مثل قائمة المشاريع في الصفحة الرئيسية)، ننفذه مرة واحدة ونحفظ النتيجة في الـ Cache لمدة معينة.

في الطلبات التالية، نقرأ من الـ Cache (سريع جداً) بدلاً من تنفيذ الاستعلام الثقيل.

// استخدام Cache::remember() - المدة 60 دقيقة $projects = Cache::remember('projects.with.tasks', 3600, function () { // هذا الكود ينفذ مرة واحدة فقط كل ساعة! return Project::with(['user', 'tasks']) ->active() ->latest() ->get(); }); // $projects الآن تحتوي على النتيجة سواء من الـ Cache أو من قاعدة البيانات

🔑 شرح Cache::remember():

  • المعامل 1 - المفتاح (Key): اسم فريد لهذا الـ Cache (مثل 'projects.with.tasks')
  • المعامل 2 - المدة (TTL): مدة الصلاحية بالثواني (3600 = ساعة واحدة)
  • المعامل 3 - Closure: الكود الذي ينفذ لاسترجاع البيانات إذا لم تكن في الـ Cache

مسح الـ Cache عند تحديث البيانات:

// عند إنشاء مشروع جديد، امسح الـ Cache Cache::forget('projects.with.tasks'); // أو أفضل: في Model باستخدام booted(): protected static function booted(): void { static::created(function () { Cache::forget('projects.with.tasks'); }); }

select(): جلب الأعمدة التي تحتاجها فقط

عند استخدام ::all()، Eloquent يجلب جميع الأعمدة (SELECT *). لكنك قد تحتاج فقط id و name و status. جلب بيانات لا تحتاجها يستهلك ذاكرة وعرض نطاق.

استخدم ->select() لتحديد الأعمدة التي تريدها فقط:

// جلب فقط الأعمدة التي نحتاجها $projects = Project::select('id', 'name', 'status', 'created_at') ->with('user:id,name') // من الـ User: فقط id و name ->active() ->get(); // في with، نستخدم: 'relation:column1,column2' // لاحظ: يجب تضمين id دائماً لأنه المفتاح الذي يربط العلاقات

مقارنة الأداء:

الطريقة البيانات المنقولة السرعة
Project::all()كل الأعمدة (8+ أعمدة)بطيء مع كميات كبيرة
Project::select('id','name')->get()عمودين فقطأسرع بمرتين

تطبيق FlowPilot: تحسين صفحة المشاريع

الآن سنطبق كل ما تعلمناه لتحسين صفحة عرض المشاريع في FlowPilot. هذه الصفحة ستستخدم بشكل متكرر ويجب أن تكون سريعة جداً.

قبل التحسين (بطيء):

// Controller - الطريقة البدائية $projects = Project::all(); // كل الأعمدة، بدون علاقات (يسبب N+1) // في Blade/React: foreach ($projects as $project) { echo $project->user->name; // N+1: استعلام لكل مشروع! echo $project->tasks->count(); // N+1 آخر! } // 100 مشروع = 1 + 100 + 100 = 201 استعلام!

بعد التحسين (سريع جداً):

// Project Model - أضف هذه الـ Scopes public function scopeWithEssential($query) { return $query->with(['user:id,name', 'tasks']) ->withCount(['tasks as completed_count' => function ($q) { $q->where('status', 'completed'); }]); } // Controller المحسّن: $projects = Cache::remember('projects.index', 3600, function () { return Project::select('id', 'name', 'status', 'deadline', 'created_at') ->withEssential() ->latest() ->paginate(20); });

النتيجة: 3 استعلامات فقط! واحد للمشاريع، واحد للمستخدمين، واحد لعدد المهام. مقارنة بـ 201 استعلام قبل التحسين. 67 مرة أسرع!

الأخطاء الشائعة في تحسين الاستعلامات

❌ الخطأ 1: نسيان ::with() في الحلقات (N+1)

أشهر خطأ أداء في Laravel! الحل: تعوّد على استخدام ::with() لكل استعلام يجلب علاقات.

❌ الخطأ 2: استخدام ::all() لجلب بيانات ضخمة

إذا كان لديك 50,000 سجل، ::all() سيحملهم كلهم في الذاكرة. استخدم ::paginate() أو ::chunk() بدلاً من ذلك.

❌ الخطأ 3: نسيان مسح الـ Cache بعد التحديث

إذا كنت تستخدم Cache، يجب مسحه عند إنشاء/تحديث/حذف البيانات. وإلا سيرى المستخدمون بيانات قديمة.

❌ الخطأ 4: جلب أعمدة كثيرة غير ضرورية

استخدم ->select() لتحديد الأعمدة التي تحتاجها فقط. ولا تنسَ جلب id في العلاقات.

❌ الخطأ 5: استخدام N+1 Queries داخل Scopes

إذا كنت تعرّف Scope يستخدم علاقات، تأكد من تمرير $query مع with(). الـ Scope لا يحمي من N+1 تلقائياً.

📝 ملخص الدرس

  • 1
    Eager Loading ::with() يجلب العلاقات مقدماً. يقلل N+1 إلى استعلامين.
  • 2
    Query Scopes دوال استعلام قابلة لإعادة الاستخدام (scopeActive, scopeStatus).
  • 3
    ::when() استعلام شرطي يضيف where فقط إذا كان الشرط محققاً.
  • 4
    Chunking يقسم النتائج الكبيرة إلى أجزاء (100 سجل كل مرة) لتوفير الذاكرة.
  • 5
    Caching يحفظ نتائج الاستعلامات الثقيلة في ذاكرة سريعة. Cache::remember() لمدة محددة.
  • 6
    select() يجلب الأعمدة التي تحتاجها فقط، مما يقلل حجم البيانات المنقولة.
